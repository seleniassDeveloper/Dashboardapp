Revisión de código del backend (Node/Express + Prisma) y del frontend (React). Hallazgos priorizados por severidad, con archivo, línea, impacto y arreglo sugerido. Los marcados VERIFICADO los confirmé leyendo el código directamente.

Lo más urgente en una frase: hoy, en producción, cualquiera puede entrar como administrador de cualquier cuenta con la contraseña "selenia" o "123456", y un usuario de un salón puede leer, editar y borrar datos (incluida información médica) de otro salón. CRÍTICO Fallos que rompen seguridad o datos hoy mismo

1. Login con contraseñas hardcodeadas = toma de control de cualquier cuenta VERIFICADO

backend/src/routes/auth.routes.js:166

const isValidPass = password === "selenia" || password === "123456" || password === prefix;

Esta ruta de login no está limitada a desarrollo: está viva en producción. Con esas contraseñas (o el prefijo del email) se genera un local-token-<userId> que el middleware acepta como admin.

Impacto: acceso total a cualquier cuenta, incluida la de dueños y super-admin.

Arreglo: encerrar todo el bloque de login local y de local-token- bajo if (process.env.NODE_ENV !== "production") y eliminar las contraseñas fijas.

2. Token de admin falsificable sin firma VERIFICADO

backend/src/middleware/requireAuth.js:10-37

Cualquier request con Authorization: Bearer local-token-<id> se autentica como ese usuario con admin: true, sin verificar nada. Además existe un token maestro por defecto: QUICK_BOOKING_TOKEN || "aura-admin-token" — si la variable no está seteada, "aura-admin-token" es una llave de admin permanente.

Impacto: admin instantáneo conociendo un id de usuario o el string fijo.

Arreglo: desactivar la ruta local-token- en producción y quitar el default "aura-admin-token" (exigir la env var).

3. Si Firebase falla, el atacante entra como admin VERIFICADO

backend/src/middleware/requireAuth.js:32, 58-59

Cuando la verificación del token de Firebase lanza error (por credenciales mal configuradas), el código cae en firebaseUser = { admin: true } en vez de rechazar. Si Firebase no está bien configurado en prod, todo request con cualquier token se vuelve admin.

Arreglo: ante fallo de verificación en producción, devolver 401 — nunca caer en una identidad admin.

4. Secretos y API keys hardcodeados en el código VERIFICADO

auth.routes.js:230 · utils/financeCrypto.js:3 · utils/consentCrypto.js:3

API key de Firebase escrita como fallback ("AIzaSy…2a8FQ"), y las claves de cifrado con defaults fijos ("finance-bypass-secret…", "consent-encryption-secret…"). Esta última cifra datos médicos/consentimiento de las clientas.

Impacto: si faltan las env vars en prod, cualquiera con acceso al repo puede falsificar tokens de finanzas y descifrar datos médicos. La API key expuesta hay que rotarla.

Arreglo: fallar el arranque si faltan los secretos en producción; nunca dejar valores por defecto; rotar la key ya filtrada.

5. Acceso entre salones (IDOR): leer/editar/borrar datos de otro tenant VERIFICADO

clients.controller.js:100,183 · crm.controller.js:12 · business.routes.js:252 · inventory.controller.js:553 · finances.controller.js:454+

Muchos handlers buscan por id sin filtrar por el salón dueño (where:{ id } en vez de where:{ id, businessId }). Casos confirmados: editar/borrar clientas de otro salón (cascada a sus turnos), leer la ficha CRM completa con alergias y notas médicas descifradas de otro salón, y GET /api/business/:id devuelve el registro completo de cualquier negocio (incluye credenciales de integraciones/SMTP/Google/WhatsApp).

Impacto: fuga y destrucción de datos entre clientes del SaaS, incluyendo PII médica. Es el riesgo legal más serio.

Arreglo: agregar businessId: req.businessId a TODOS los where por id (o verificar record.businessId === req.businessId antes de leer/mutar).

6. El "un clic = 3 cosas" solo hace 1: no registra venta ni comisión VERIFICADO

backend/src/controllers/appointments.controller.js → finalizeAppointment

Al finalizar un turno, el handler solo pone status:"DONE", guarda nota/fotos y descuenta stock. No crea ningún registro de venta en finanzas ni calcula la comisión del profesional. Peor: las finanzas se "reconstruyen" al leer, con una comisión fija del 40% (finances.controller.js), ignorando el commissionType/commissionValue configurado por servicio.

Impacto: no hay un libro de ventas auditable, las comisiones mostradas son inventadas, y la "ganancia neta" no es real. Justo lo que vendés en la demo (venta + comisión + stock en un clic) hoy no queda persistido.

Arreglo: en finalizeAppointment, crear el registro de venta y el de comisión (usando la config del servicio y el finalPrice real) dentro de un prisma.$transaction junto al cambio de estado y el descuento de stock.

7. Descuento de stock sin transacción, sin idempotencia y con el error silenciado VERIFICADO

appointments.controller.js:~813-882 (bloque de inventario)

El descuento de inventario corre siempre que se llama al endpoint (no está protegido por "ya estaba DONE"), fuera de una transacción, y dentro de un try/catch que solo hace console.error y igual devuelve éxito (200).

Impacto: tocar "finalizar" dos veces (o un reintento del navegador) descuenta el stock dos veces; si falla a la mitad, el turno queda DONE con inventario a medias y el sistema reporta OK. Corrupción silenciosa de stock.

Arreglo: if (appt.status === "DONE") return … al inicio; envolver estado + nota + inventario en un $transaction; no tragarse el error.

ALTO Bugs de lógica, dinero y accesos

8. Webhooks de pago (Mercado Pago / Stripe) sin verificar firma + modo "mock" activable

services/billing/mercadopago.provider.js:79 · stripe.provider.js

verifyWebhook hace return true. Si el token del proveedor no está seteado o empieza con mock_, se confía en datos del atacante y se puede poner cualquier suscripción en active sin pagar.

Arreglo: validar la firma real del webhook y deshabilitar por completo el modo mock cuando NODE_ENV === "production".

9. Aprobación de suscripción pública y sin auth

billing.controller.js:372 → GET /api/billing/quick-approve

Con solo un approvalToken (que viaja en la URL, queda en logs) se pone el negocio en plan active. Sin auth ni verificación de fuerza del token.

Arreglo: exigir auth + super-admin, o token de un solo uso, alta entropía, corto y por POST.

10. Frontend: la URL del backend está hardcodeada e ignora la variable de entorno

dashboard-react/src/lib/api.js:5-14

normalizeBaseUrl() devuelve "https://dashboard-api-r6j9.onrender.com/api" antes de leer VITE_API_URL. En prod la env var no tiene efecto.

Impacto: no podés repuntar el frontend a otro backend/entorno sin tocar código; el banner de seguridad de API queda muerto.

Arreglo: priorizar import.meta.env.VITE_API_URL y usar la URL fija solo como fallback.

11. Frontend: la pantalla de "Acceso No Autorizado" es código muerto VERIFICADO por el subagente

auth/AuthProvider.jsx (value del useMemo) · LoginGate.jsx:19

isUnauthorized está en las dependencias del useMemo pero no se incluye en el objeto value que expone el contexto, así que en LoginGate siempre es undefined.

Impacto: usuarios inactivos/pendientes/rechazados pueden colarse porque la guarda principal nunca se activa.

Arreglo: agregar isUnauthorized (y firestoreError) al objeto que retorna el contexto.

12. Finanzas cuenta como ingreso turnos CONFIRMED (no realizados) y usa el precio de catálogo

finances.controller.js:53,54,150,206

Suma DONE || CONFIRMED como ingreso realizado y usa service.price en vez del finalPrice realmente cobrado.

Impacto: ingresos, comisiones y ticket promedio inflados; descuentos/recargos nunca se reflejan.

Arreglo: contar solo DONE y usar Number(a.finalPrice ?? a.service?.price ?? 0).

13. Métodos de pago del panel financiero son inventados por índice del array

finances.controller.js:74-78, 204

Reparte los ingresos en métodos de pago con idx % 6 en vez del paymentMethod real que sí se guarda al finalizar.

Arreglo: agregar por el appointment.paymentMethod real.

14. Página pública de reservas crashea si la API no devuelve un array

booking/BookingForm.jsx:100-137

setServices(dataServices) directo desde res.json() y luego services.map(...). Si el endpoint devuelve un objeto de error con 200, la página pública queda en blanco.

Arreglo: setServices(Array.isArray(dataServices) ? dataServices : []) (y lo mismo con profesionales).

MEDIO Bugs que fallan en ciertos casos

15. Drift entre schema.prisma y las migraciones → 500 si se usa migrate deploy

backend/prisma/schema.prisma vs prisma/migrations/

El schema define ~51 modelos pero las migraciones solo crean 7 tablas. Campos como Business.industry, Business.model, Business.appointmentStatuses no tienen migración. Hoy "funciona" porque server.js usa prisma db push; cualquier entorno con migrate deploy se rompe (este es el origen del 500 del cambio de rubro).

Arreglo: generar una migración baseline con el schema completo y pasar prod a migrate deploy, o documentar db push como estrategia oficial.

16. seedBranchesIfNeeded() usa req fuera de alcance → ReferenceError

finances.controller.js:6 (función sin parámetro req)

Referencia req.businessId dentro de una función que no recibe req. El error se traga en un try interno, así que las sucursales nunca se siembran y cada carga del dashboard loguea error.

Arreglo: pasar businessId como parámetro: seedBranchesIfNeeded(businessId).

17. Rutas montadas sin checkTenant → businessId undefined

app.js:125 (/api/dashboard) · app.js:137 (/api/invitations) · app.js:149 (/api/mobile-scans)

Sin checkTenant, req.businessId es undefined y las queries matchean businessId IS NULL (widgets rotos / datos en el "bucket null"). mobile-scans además no tiene auth y usa un buffer global compartido entre todos los tenants.

Arreglo: montar esas rutas con requireAuth + checkTenant y aislar el buffer por negocio.

18. Desincronización stock global vs. lotes (permite oversell oculto)

appointments.controller.js:~813-840

Los lotes se descuentan FIFO por un lado y el stock global del producto se setea con Math.max(0, stock - qty) por otro. Si no alcanza, el Math.max(0,…) esconde el faltante en vez de bloquear.

Arreglo: derivar el stock del producto de la suma de lotes y rechazar/avisar si no hay disponible.

19. createService tira 500 si falta name del todo

services.controller.js:13,109

El validador solo rechaza name vacío, no ausente; después name.trim() explota → 500 en vez de 400.

Arreglo: if (!name?.trim()) return res.status(400)…

20. Idempotencia del webhook de pago fuera de la transacción (race)

billing.controller.js:223 vs 272-281

Dos webhooks idénticos concurrentes pasan el chequeo y uno rompe la constraint única dentro de la transacción, abortando una activación válida.

Arreglo: usar la constraint única vía create/upsert y tratar P2002 como "ya procesado".

21. Frontend: token de Firebase con force-refresh en cada request + JSON.parse sin try/catch

AuthProvider.jsx:521 · BatchControl.jsx:44 · FinancialReports.jsx:94

getIdToken(true) fuerza refresco en cada llamada (latencia/rate-limit). Y hay JSON.parse(localStorage…) sin protección que crashea el componente si el dato está corrupto.

Arreglo: usar getIdToken() (cacheado) y solo forzar en reintento por 401; envolver los JSON.parse en try/catch con fallback.

BAJO Mejoras y riesgos menores

22. Super-admins hardcodeados e irrevocables

utils/superadmin.js:3,13 · appointments.controller.js:464 · health.routes.js · tenant.middleware.js:30

seleniadeveloper@gmail.com y selenisdeveloper@gmail.com devuelven true aunque se setee la env var — god-mode permanente en todos los tenants, no revocable por config.

Arreglo: manejar super-admin solo desde env/DB; quitar los literales.

23. Config de entorno solo advierte, nunca falla en producción

config/env.js:27-54

Faltar DATABASE_URL, credenciales de Firebase o incluso AUTH_DISABLED=true solo hace console.warn. Combinado con los críticos 2-4, un deploy mal configurado corre en modo inseguro sin avisar.

Arreglo: fallar el arranque ante secretos críticos faltantes en producción.

24. Endpoint de health filtra flags internos

health.routes.js:17-20,37

Expone públicamente authDisabled y flags de super-admin — ayuda de reconocimiento para un atacante.

Arreglo: no exponer esos flags en el endpoint público.

25. Webhook de WhatsApp sin verificar firma de Meta

routes/whatsapp.routes.js:15

Acepta y loguea cualquier body sin validar X-Hub-Signature-256. Bajo impacto hoy (solo loguea) pero es un punto de ingreso abierto.

Arreglo: validar el HMAC con el app-secret de Meta.

26. Efectos secundarios "fire-and-forget" que fallan en silencio

appointments.controller.js:674-679 · billing.controller.js:362

Recordatorios, workflows y emails de activación corren con .catch(console.error) — si fallan, desaparecen sin reintento ni aviso.

Arreglo: cola con reintentos o al menos registro visible del fallo.

27. Cálculos de fechas sin zona horaria + logout de inactividad muy agresivo

crm.controller.js:63 (server-local time) · App.jsx:36 (logout a los 3 min)

Los umbrales "hace 30-60 días" usan hora local del servidor (puede errar por un día). El auto-logout por inactividad a los 3 minutos es muy corto para un panel de trabajo.

Arreglo: usar la zona del negocio para fechas; subir el timeout de inactividad (p. ej. 30-60 min).

  2. Esta semana (integridad de datos): reescribir finalizeAppointment para registrar venta + comisión real y descontar stock, todo en un $transaction con guarda de idempotencia (#6, #7). Corregir finanzas: solo DONE, finalPrice y método de pago real (#12, #13).

  5. Limpieza: super-admins por config, env que falle en prod, y el resto de los bajos.
